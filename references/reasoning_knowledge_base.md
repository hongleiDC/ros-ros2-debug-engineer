# 推理、公式与代码逻辑知识库

## 目的

项目知识库不仅保存“结论”，还要保存结论如何得到、采用了哪个公式版本、每个数学符号对应哪一个代码变量、单位和坐标系是否一致，以及哪些测试真正验证了这条逻辑。

这套记录用于回答四个问题：

1. 代码实现的数学模型是什么；
2. 每一步推理是否有已验证前提和明确规则；
3. 公式符号、配置参数、消息字段和代码变量是否严格对应；
4. 修改代码后，旧推理、公式、变量映射和测试是否已经失效。

## 知识目录

采用项目知识库后，维护以下目录：

```text
project_knowledge/
├── formulas/             # FORM-*：公式、符号、假设、约定和逐步推导
├── variable_mappings/    # MAP-*：公式符号到具体代码变量的实现映射
├── reasoning_chains/     # REAS-*：从条件到结论的可审计推理链
└── audits/               # AUD-*：逻辑、单位、映射和证据审计结果
```

### FORM 公式记录

每一个实际影响代码行为的公式都要有独立版本。公式记录至少包含：

- 公式的 LaTeX 表达式和用途；
- 假设及其验证状态；
- frame、变换方向、矩阵乘法顺序、时间基准和索引顺序；
- 每个 `SYM-*` 的含义、角色、量纲、标准单位、frame、方向、时间基准、形状和有效范围；
- 逐步推导 `DER-*`，每一步说明操作和依据；
- 必须映射到代码的符号；
- 测试、来源、适用 commit 和证据。

公式语义变化时必须创建新版本。不得在保留旧版本号的情况下改变符号含义、单位、方向或状态排列。

### MAP 公式—代码变量映射

每个公式在每个实现位置都要有独立映射记录。映射表必须覆盖：

| 内容 | 要求 |
|---|---|
| 数学符号 | 必须是 FORM 中已定义的 `SYM-*` |
| 代码变量 | 精确到 identifier，不得只写“时间变量” |
| 代码位置 | 仓库、commit、package、文件、函数/类、行范围 |
| 物理语义 | 必须与 FORM 完全一致 |
| 单位 | 分别记录公式单位和代码存储单位 |
| 单位转换 | 单位不同时记录表达式、方向和测试 |
| frame/方向 | 例如 `T_map_base` 的 from/to 语义 |
| 时间基准 | sensor、host、ROS time、steady clock 等 |
| 形状和索引 | 向量维度、状态顺序、矩阵行列含义 |
| 类型 | C++/Python 类型、消息字段或配置参数类型 |
| 证据 | 代码、测试、日志或手算样例 |

一个关键代码变量只能代表一个物理量。相同 identifier 若对应不同单位、frame、方向、时间基准或形状，审计必须失败。

### REAS 推理链

每一个用于决定代码修改、确认根因或宣布成功判据满足的关键结论，都要记录为推理链：

```text
已验证条件 COND-* / 明示假设 ASM-*
        ↓
STEP-1：使用的定义、公式或软件语义
        ↓
STEP-2：代数、坐标、统计或代码路径变换
        ↓
STEP-n：证据、反例与边界检查
        ↓
结论：支持哪个 SC-*，仍有哪些限制
```

每一步必须包含：前提引用、规则类型、规则或定义、具体操作、变化前后表达式、涉及的符号和代码变量、证据、输出、验证状态以及反例检查。

以下情况不得把推理标记为 `verified`：

- 已知条件或假设仍是 unknown/candidate；
- 任一步骤尚未 verified；
- 结论没有直接证据；
- 仍存在 unresolved 项；
- 引用的公式或映射版本已过期；
- 推理步骤引用未来步骤或不存在的前提。

## 工作流程

### 代码审查或调试前

1. 读取活动 `GOAL-*` 和相关 `SC-*`。
2. 搜索 `FORM-*`、`MAP-*`、`REAS-*` 历史记录。
3. 对公式相关代码运行：

```bash
python3 scripts/logic_audit.py /path/to/project_knowledge \
  --workspace /path/to/repository \
  --audit-id AUD-0001
```

4. 若公式、映射或推理记录为空，默认审计失败。确认项目确实没有公式相关逻辑时才使用 `--allow-empty`。
5. 审计失败时先修复知识与代码的不一致，不得直接继续改参数或宣称找到根因。

### 编码过程中

1. 新公式先建 FORM，再写代码。
2. 每加入一个关键变量，同时加入或更新 MAP。
3. 每个关键判断、转换和状态更新写入 REAS 推理链。
4. 代码注释引用 `FORM-*`、`DER-*`、`MAP-*` 或 `REAS-*`，但知识库仍是完整来源，不能只靠注释。
5. 变量重命名、单位转换、frame 方向、状态顺序和公式版本变化必须同步更新知识记录和测试。
6. 不允许先写代码后凭结果反向编造推导。

### 代码修改后

使用严格模式生成审计报告：

```bash
python3 scripts/logic_audit.py /path/to/project_knowledge \
  --workspace /path/to/repository \
  --audit-id AUD-0002 \
  --write-report \
  --strict-warnings
```

只有审计无 error，且任务要求严格时无 warning，才能进入构建、bag 回放或验收阶段。审计通过只证明记录之间结构和语义一致，不自动证明模型本身真实正确；仍需手算样例、单元测试、数据实验和回归证据。

## 自动审计范围

`logic_audit.py` 检查：

- 公式、映射、推理和审计 Schema；
- FORM/MAP/REAS/GOAL 交叉引用；
- 公式版本是否一致；
- 必须实现的符号是否有代码映射；
- 映射语义是否与公式符号完全一致；
- 同一 identifier 是否出现冲突语义；
- 变量名和单位后缀是否明显冲突；
- 不同单位间是否记录转换表达式和测试；
- 源文件、identifier 和行范围是否仍存在；
- 推理步骤是否按拓扑顺序引用前提；
- verified 推理是否仍含未知前提、未验证步骤或未解决问题；
- 关键变量是否使用 `tmp`、`val`、`data2`、`x1` 等无语义名称。

自动审计不能代替数学证明、真实硬件验证或领域专家判断。它负责发现可追溯性断裂和高概率实现错误，并强制 Agent 对无法证明的部分保持 `unknown` 或 `candidate`。

## 安全登记

先在临时文件中编写记录，再使用：

```bash
python3 scripts/register_reasoning_knowledge.py \
  /path/to/project_knowledge /tmp/FORM-0001.yaml \
  --reason "Document the timestamp correction model before implementation"
```

该工具按 Schema 验证、检查全知识库交叉引用、拒绝覆盖 verified/deprecated 记录，并以原子方式写入知识库。替换 draft/candidate 记录时必须显式使用 `--replace-draft`。
