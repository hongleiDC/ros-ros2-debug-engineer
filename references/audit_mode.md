# ROS 审计模式

## 触发条件

审计由“风险 + 请求动作”触发，不由领域名称自动触发。

进入 `audit` 的典型情况：

- 用户明确要求公式、变量、实验和结论可追溯；
- 准备修改、部署或验收可能造成物理风险的控制/硬件逻辑；
- 准备修改关键标定、时间同步、状态估计或误差传播逻辑，且错误难以由普通测试隔离；
- 多人长期协作、跨会话实验或正式质量流程需要持久证据。

保持 `debug` 的情况：概念解释、只读代码审查、局部公式核对、普通编译/launch/QoS/topic/参数问题，以及尚未进入修改或部署阶段的标定和状态估计咨询。

## 最小审计链

按风险选用，不机械创建全部记录：

1. [目标管理](goal_management.md)：目标、成功判据和边界；
2. [实验管理](experiment_management.md)：可重复条件与结果；
3. [公式与变量追溯](formula_variable_traceability.md)：真正影响输出的关键公式；
4. [推理知识库](reasoning_knowledge_base.md)：需要跨会话长期维护时；
5. `logic_audit.py`：验证记录一致性。

## 原则

- 记录服务于决策和回归，不为记录而记录；
- checkpoint 只发生在目标变化、关键实验、重要补丁或交接时；
- 审计不能替代测试、bag、仿真或硬件证据；
- 用户未授权时不向目标仓库写入知识记录；
- 即使进入审计，也继续优先最小代码面、最少假设和最小修复。

## 交付

在普通调试交付之外，只补充实际使用的目标、公式/变量、实验、审计状态和未验证边界。
