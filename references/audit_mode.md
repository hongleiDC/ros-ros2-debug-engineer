# ROS 审计模式

## 何时使用

仅在以下情况默认启用完整记录：

- 控制、安全关键硬件或可能造成物理风险；
- 标定、坐标变换、时间同步、状态估计和误差传播；
- 正式验收、法规/质量流程或多人长期维护；
- 复杂问题需要多轮实验并可能跨会话继续；
- 用户明确要求公式、变量、实验和结论可追溯。

普通编译、launch、QoS、topic、参数或单点逻辑错误不要升级到审计模式。

## 最小审计链

按实际风险选择，不机械创建全部记录：

1. 使用 [目标管理](goal_management.md) 保存目标和成功判据；
2. 使用 [实验管理](experiment_management.md) 保存可重复条件；
3. 公式关键时使用 [公式与变量追溯](formula_variable_traceability.md)；
4. 需要长期知识时使用 [推理知识库](reasoning_knowledge_base.md)；
5. 使用 `logic_audit.py` 验证记录一致性。

## 执行原则

- 记录服务于决策和回归，不为记录而记录；
- 只为真正影响输出的公式建立映射；
- checkpoint 发生在目标变化、重要实验、关键补丁或上下文交接时，不按固定工具调用次数触发；
- 仍优先定位根因和最小修复；审计不能替代测试、bag、仿真或硬件证据；
- 用户未授权时，不向目标仓库写入知识记录。

## 交付

审计任务在普通调试交付之外补充：

- 目标与成功判据；
- 公式、变量、单位、frame 和时间基准；
- 实验条件、结果和产物；
- 逻辑审计状态；
- 尚未验证的假设和适用边界。
