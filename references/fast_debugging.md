# ROS 快速调试

## 目标

用最少代码阅读和最少命令，把问题收敛到一个可验证根因。不要把诊断过程变成知识库建设项目。

## 紧凑状态

整个任务只维护一小段状态：

```text
环境: ROS 2 Humble / Ubuntu 22.04 / CycloneDDS
范围: localization_bringup -> imu_filter
现象: /imu/data 有发布，滤波节点无输出
首要假设: 订阅 QoS 与发布端不兼容
下一检查: 查看端点 QoS
```

信息确认后更新状态，不重复叙述历史。

## 故障分层

按最早可能失败的层级向下定位：

| 层 | 典型现象 | 首选证据 |
|---|---|---|
| 构建 | 找不到包、符号、接口或依赖 | 完整首个错误、package/CMake、overlay |
| 启动 | 节点未启动、参数不生效 | launch 展开、实际参数、namespace/remap |
| 图连接 | 节点存在但端点未匹配 | node/topic/service/action info |
| 通信 | 有端点但无数据或丢数据 | 类型、QoS、RMW、网络、频率 |
| TF/时间 | extrapolation、跳变、不同步 | frame 树、stamp、clock、缓存窗口 |
| 调度/资源 | 延迟、卡死、掉帧 | executor、callback group、阻塞、队列、CPU |
| 数据/算法 | 数值异常、漂移、发散 | 输入分布、单位、边界条件、算法不变量 |

如果上层尚未证明正常，不直接跳到算法参数调优。

## 假设排序

最多保留三个假设：

1. **最可能**：结合现象、最近改动和 ROS 高频故障；
2. **次可能**：能够解释大部分现象；
3. **低概率高风险**：会导致严重安全或数据破坏。

优先级可用以下思路判断：

```text
价值 = 区分能力 × 根因概率 ÷ 检查成本
```

不要把所有 ROS 知识都列成“可能原因”。

## 高信息增益检查

### 构建

- 只分析第一个真实编译错误，后续错误往往是级联结果。
- 核对 source/build/install overlay 顺序和环境是否重新 source。
- 对接口生成，核对 `rosidl_generate_interfaces`、依赖导出和目标链接。

### 启动与参数

- 核对实际节点名、namespace、remap 后的最终接口名。
- 核对参数是否声明、类型是否匹配、参数文件键是否对应完整节点路径。
- 对 lifecycle 节点核对当前状态，不把“进程存在”当成“节点 active”。

### 通信

- 先比较消息类型和端点 QoS，再怀疑 DDS 本身。
- 传感器流重点检查 reliability、durability、history、depth 和 deadline。
- 多机问题先确认 domain ID、RMW、组播/发现服务器和防火墙边界。

### TF 与时间

- 明确所需变换方向和查询时刻。
- 核对 stamp 来源：硬件采样、驱动接收、系统时钟、ROS time 或仿真 clock。
- 区分“变换不存在”“变换太旧”“请求来自未来”“外参方向错误”。

### 并发与性能

- 查找回调中的磁盘、网络、service、sleep、锁等待和重计算。
- 检查 mutually-exclusive callback group 是否把互相依赖的回调串死。
- 检查队列是否无界、是否持续积压，以及数据是否在 composition 边界重复复制。

### 算法

- 先核对单位、frame、符号方向、索引和时间对齐。
- 用最小手算样例或固定输入测试一个不变量。
- 参数调节不能替代输入错误、坐标错误或时间错误的修复。

## 代码阅读预算

初始只读取：

- 与错误直接相关的 `package.xml` 或构建文件；
- 一个 launch/参数文件；
- 发布端与订阅端的关键实现；
- 必要时一个消息定义或算法核心文件。

只有当前证据无法区分活动假设时才扩大范围。不要在普通单点故障中先运行全仓扫描。

## 修改与验证

修改前用一句话说明：

```text
本补丁通过统一发布/订阅 QoS，验证“端点已发现但策略不兼容”这一根因。
```

修改后优先运行最小验证：

- 单目标构建；
- 单元测试；
- 单个 launch；
- 短时 topic/TF/参数检查；
- 同一 bag 的前后对比。

只有最小验证通过后，再决定是否需要更大范围回归。

## 停止规则

- 一个根因能解释所有关键现象，并且修复后的同一条件不再复现：停止。
- 发现架构债务但与当前故障无直接关系：单独列为后续建议，不在当前补丁中扩展。
- 证据不足：只提出下一项最有区分度的检查。

默认交付格式：

```text
根因
证据
修改
验证
剩余风险
```
