## 纯算法核心与 ROS 适配层

推荐：

```text
core library
  input/output domain types
  deterministic update/process APIs
  no parameter server, TF buffer, publisher or clock globals

ROS adapter
  message conversion
  parameters
  TF lookup
  QoS and publishers/subscribers
  lifecycle and diagnostics
```

收益：算法可离线、可回放、可 fuzz、可移植，ROS 发行版变化被限制在适配层。

例外：高度依赖 ROS graph 或动态参数的基础设施节点可以直接使用 ROS API，但仍要隔离业务逻辑。

## 进程、节点与组件

### 独立进程

适用于：

- 硬件驱动和第三方不稳定库；
- 安全监控和控制边界；
- 需要独立重启、权限或资源限制；
- 不同主机或容器部署。

代价：序列化、发现和进程管理成本。

### Component composition

适用于：

- 图像、点云等大消息高频流水线；
- 组件由同一团队维护并共享故障域；
- 需要 intra-process 或更低延迟。

代价：一个崩溃可影响整个容器，线程和 callback group 设计更关键。

### 插件

适用于：

- 同一稳定接口下切换算法、传感器模型或规划器；
- 运行时或部署时选择实现。

插件接口必须小而稳定，禁止把整个 ROS node API 暴露为插件契约。

## 流水线与共享状态

### Topic 流水线

适用于松耦合、异步、可观测的数据转换。每个阶段要定义时间戳、frame、队列和过期策略。

### 进程内流水线

适用于极低延迟和大数据，但需要明确线程、所有权和可观测边界。不要为了零拷贝牺牲所有故障隔离。

### 共享黑板/全局状态

只在确有多读者一致快照需求时使用。优先不可变快照、版本号和明确写入者；避免多个 callback 隐式修改全局对象。

## 接口模式

- 连续数据：topic；
- 快速、幂等查询或设置：service；
- 长时、可反馈、可取消任务：action；
- 静态配置：版本化文件或参数；
- 运行模式：显式状态机，不依赖多个布尔参数组合；
- 健康信息：diagnostic/status topic 与结构化错误码。

接口包应避免依赖实现包。自定义消息只在标准消息无法表达稳定语义时创建。

## 状态与生命周期

建议 lifecycle 的对象：

- 设备驱动；
- 需要加载模型/地图/标定的组件；
- 启动顺序敏感的估计和控制节点；
- 需要明确 safe activation 的硬件接口。

不建议仅为形式把所有简单无状态转换节点改为 lifecycle。

系统级 orchestrator 应掌握依赖图和恢复策略。节点只报告本地状态，不各自实现一套全局启动逻辑。

## 多机与部署

- 用 namespace 表示机器人/站点实例，而不是写死 topic 前缀；
- 明确 ROS domain、RMW、发现机制、网络 ACL 和时钟同步；
- 高带宽原始数据尽量在采集主机处理，跨机传输结果或压缩数据；
- 控制链路与非关键可视化链路分开评估；
- 容器边界与故障/资源边界一致，不要一个节点一个容器；
- 配置、镜像、接口和 bag 数据均需版本标识。

## 反模式

### 节点过度拆分

症状：大量微小节点、复杂 launch、重复序列化、调试困难。修复：按故障域、发布节奏和数据复制成本重新聚合。

### 巨型节点

症状：驱动、算法、TF、控制和日志耦合，无法测试或独立重启。修复：抽离算法核心和硬件边界。

### Topic 作为远程函数调用

症状：请求/响应配对、超时和并发语义靠自定义 ID 拼装。修复：使用 service 或 action。

### 参数作为动态状态

症状：高频改参数驱动状态机、无法追踪竞争更新。修复：显式命令、状态消息或 action。

### 多个 TF 发布者

症状：frame 跳变、树不稳定、重定位混乱。修复：为每条 transform 指定唯一所有者和切换协议。

### 默认 QoS 到处复制

症状：跨网络丢包、端点不匹配、缓存不可控。修复：按数据语义和失败成本定义 QoS 契约。

### 在 callback 中阻塞

症状：service 调用、磁盘写入或锁等待阻塞其他回调。修复：隔离 callback group、后台工作线程或异步接口，并设置有界队列。
