# ROS 架构决策模式

本文件只用于边界取舍，不重复完整架构流程。

## 运行边界决策

| 方案 | 适合 | 主要收益 | 主要代价 |
|---|---|---|---|
| 独立进程 | 驱动、控制、安全监控、不稳定第三方库、独立部署 | 故障与权限隔离、独立重启 | 序列化、发现、进程管理 |
| Component composition | 图像/点云高频流水线、同团队、同故障域 | intra-process、低延迟、少复制 | 崩溃共享、线程设计更难 |
| 插件 | 稳定接口下切换算法或设备模型 | 实现可替换、部署灵活 | ABI/API 稳定性要求 |
| 普通库 | 确定性算法、转换、优化和模型 | 易测试、易复用、无 ROS 耦合 | 需要适配层 |

默认先将算法做成普通库，再决定 ROS 运行边界。

## 接口选择

| 需求 | 选择 | 不要选择 |
|---|---|---|
| 连续异步数据 | topic | service 轮询 |
| 快速幂等查询/设置 | service | 自定义 request topic |
| 长时、反馈、取消 | action | 阻塞 service |
| 静态版本化配置 | 文件/参数 | 高频参数更新 |
| 模式与系统状态 | 显式状态机 | 多个布尔参数组合 |
| 健康与故障 | diagnostics/status + 错误码 | 仅自由文本日志 |

## 生命周期决策

建议 lifecycle：设备驱动、加载地图/模型/标定的组件、启动顺序敏感的估计和控制节点、需要 safe activation 的硬件接口。

不建议：简单无状态转换节点，或没有明确配置/激活/恢复语义的组件。

## 多机与部署决策

- 用 namespace 表示机器人或站点实例，不写死前缀；
- 明确 ROS domain、RMW、发现机制、网络 ACL 和时钟同步；
- 原始高带宽数据尽量在采集主机处理；
- 控制链路与可视化/记录链路分开评估；
- 容器边界对齐故障和资源边界，不要一个节点一个容器；
- 配置、镜像、接口和 bag 均带版本。

## 常见反模式

| 反模式 | 症状 | 修复方向 |
|---|---|---|
| 节点过度拆分 | launch 复杂、重复序列化、难追踪 | 按故障域、频率、复制成本聚合 |
| 巨型节点 | 驱动、算法、TF、控制耦合 | 抽离算法核心和硬件边界 |
| Topic 模拟 RPC | 自定义 ID、超时和并发混乱 | service/action |
| 参数作为动态状态 | 竞争更新、不可追踪 | 显式命令与状态机 |
| 多个 TF 发布者 | frame 跳变、树不稳定 | 唯一所有者和切换协议 |
| 默认 QoS 复制 | 端点不匹配、缓存不可控 | 按语义和失败成本定义契约 |
| callback 阻塞 | 卡顿、饥饿、掉帧 | 隔离 I/O、异步工作、有界队列 |
| 全局共享可变状态 | 隐式竞争、不可重放 | 唯一写者、不可变快照、版本号 |

## 快速决策规则

1. 需要独立重启或权限隔离：独立进程。
2. 大消息、高频、同故障域：考虑 composition。
3. 只替换算法实现：插件或普通库，不新增通信层。
4. 需求不能量化：先给假设和验证计划，不武断选线程、QoS 或队列。
5. 一个架构决定必须能对应某个约束、风险或可测收益。
